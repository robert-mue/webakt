Overview

This reference manual is organised into three main sections:

  • Concepts: Underlying concepts
  • Menus: the menus at the top of the screen, and their commands
  • Panels: the panels that appear in the webAKT workspace, which display aspects of the knowledge base and allow the user to interact with it.

Concepts

Boolean search

Collection

An AKT knowledge base consists of a number of components - the metadata, statements, sources, formal terms, etc. All these components, with the exception of the metadata, have a number of similar items. I'll call these collections of items.

When the user interacts with the knowledge base - to view it, or to add or modify it - much of this interaction involves working one or other of the collections. In webAKT, these are primarily accessed via the KB menu: e.g. the Statements command or the sources command.

When you do this, webAKT opens up a window which show you a scrollable list of all the items in the collection. This list is actually a table, with the columns being the ID for each item and possibly other properties. For example, for statements, it might show the formal and English versions of each statement.

Alternative...

An AKT knowledge base (AKT5 or webAKT) consists largely of collections of similar items - sentences, sources, formal terms, etc.

In AKT5, each of these is implemented separately: the window for displaying the collection of statements is separate from the window for displaying sources.

In webAKT, there is a single window type (a panel in webAKT speak) for displaying a collection, based on the premise that the operations the user wants to be able to do is the same for each collection: display its items, filter this display, create a new item, and display a selected item. This means that the user has a similar experience in viewing any collection; it reduces the amount of code that needs to be maintained; and it simplifies the task of adding a new collection (e.g. "location") should the need arise.

Diagram

Filter

When you go shopping on the web, it is very common to be able to filter some product according to some criteria (size, colour, price...). It would be great if we could do the same thing for items in a collection.

In AKT5, the standard window for displaying a collection does not enable you to do this. [Check] Instead, one would need a User Tool, which may or may not have been developed, and would probably be limited in scope.

In webAKT, a filter mechanism has been implemented which closely matches the ones you see in a web store. For each collection, there are a number of categories (corresponding to colour, size, price...), with each category having a number of options. The user can choose one or more options from each category (or not use a category), and can choose one r more categories. Options with a category are OR'd together, and categories are AND'd together (just like in a web store).

The same filter mechanism is available for all collections (it is implemented once, not for each collection), though the actual set of categories and options differs for each collection (the filters are different for statements than they are for sources).

Formal term

Statements are made up of two types of words. The first are keywords which are part of the AKT language, such as “att_value”, “if” and “causes1way”. The others are “formal terms”: words provided by the original source (perhaps moderated by the investigator) relating to specific aspects of their knowledge, such as the names of plants (such as “maize”, the names for attributes (such as “height”), and the values that attributes can take (such as “high”). In general, these are nouns, except for the values of attributes, which can be adjectives.

For the knowledge base to be useful, It is important that formal terms are consistent, and as unambiguous as possible. However, different users may have a different understanding or concept of the same term. For example ‘alley cropping’ may suggest to some a system where the tree component is made up of a single or multiple row of trees, whilst others envisage a system of dense hedgerows. For this reason it is useful to be able to define exactly what is meant by the term in the context of a specific knowledge base. This is the role of the “Definition” field for a formal term.

Conversely, there may be several words used for the same underlying concept, such as the local, English and scientific names for a plant, and even different local and English names. In order to cater for these, AKT5 (and hence webAKT) supports the notion of synonyms. Please see the separate note in the webAKT Reference Manual (Concepts) on Synonym.

Hierarchy

Image

Images were very difficult to work with in AKT5. They had to be bitmaps (who uses a bitmap these days?), and naming was tricky.

In webAKT, images can be readily added simply by providing its URL. This can either link to an image anywhere on the web, or to a file on one’s own PC.

Making an image and adding it to a knowledge base can be done very easily. You can take a photo in the field, upload it to e.g. Instagram, and reference it in the KB, all in a matter of minutes.

Moreover, images take up no space in the KB file – all that is stored in the file is its URL.

Conversely, each item can be associated with multiple images, if the knowledge-base originator decides that would be useful. These multiple images could either be displayed using a carousel-type display 9as currently happens in the formal_term_details panel), or could all be displayed 9in, say, a custom tool).

Like memos in webAKT, the image ID (not the image itself) is simply a property of the item it is associated with. This means that the same image can be associated with more than one item. However, unlike memos, but like statements, sources etc, images are managed as a collection, and have their own entry in the KB menu. However, unlike all the other collections, images have their own panel for displaying them, since the tabuated listbox style used for oter collections is not appropriate for displaying thumbnails of the images in the collection.

Currently, only formal terms can have an image associated with them, but it would be easy to allow any type of item (e.g. a statement or a source) to have one or more, images associated with it.

Listbox

The central element of the window (=panel) for displaying a collection is a scrollable list of all the items in the collection.

In AKT5, this uses a built-in "control" (a user-interface component) which can only have a single term in it - basically the identifier for each item.

In webAKT, this has been replaced by a "listbox": a table, with one row for each item. The columns in the table display the various properties for each item - e.g. for a statement, this could be its ID, its type (att_value or causal), and its formal and English versions. Which columns are displayed is under both program and user control, so for example the table could be a simple list of IDs (as in AKT5) if that is what is wanted in a particular context.

This radically improves the user's ability to work with collections. Not only can they use the list to select an item, but they can view whatever properties they want for the whole collection (or a subset of it - see filters below).

Memo

A memo is a note attached to a particular item - a statement or a source, for example - explaining something about it. For example, a source might have a memo giving additional information about the source, or a statement might have a memo explaining why the investigator chose a particular formal representation of the original statement.

Memos are handled in a quite different way in webAKT compared with AKT5. In webAkT, a memo is simply a property of the particular item it is attached to, whereas in AKT5 a memo was a distinct type of item (forming a collection, in webAKT's terminology).

The reason for this change is two-fold. First, in database-design terms, a memo really does seem to be like a property of an item, so why not simply add an extra field to the item to hold the memo (like any other property it has)? Second, it is hard to see why any database user should want to see the "collection of memos", unlike a collection of sources or statements, since memos (at least, most memos...) only have meaning in relation to the particular item they are associated with.

There is another difference between webAKT and AKT5 which is worth noting in passing. In AKT5, a special memo is used to capture model metadata. This is done by giving this one memo (and no others) various fields (for project purpose, location etc), which are then used to populate the window giving background information (aka metadata) about the knowledge base. This difference is not noticeable by the user, but does in my opinion hide the importance of recognising knowledge base metadata as an important concept in its own right.

Metadata

The metadata for a knowledge base contains information about the knowledge base, such as its purpose, and the name of the investigators involved in the project.

In AKT5, the metadata information was simply held as a memo, albeit one which had various subfields for the different types of metadata. In webAKT, the metadata is another top-level component, like statements and sources, the main difference being that, unlike the others, it is not a collection (containing various items), but a single item with various properties (fields).

The following are the fields that constitute the metadata for a knowledge base:

  • Title of this knowledge base: – Here the full title of the knowledge base should appear, for example, the full title of the ‘treefodd’ knowledge base is ‘Tree Fodder and Indigenous Knowledge in Nepal’.
  • What the knowledge base is about: – Here the area of local knowledge covered by the elicitation process should be described.
  • This knowledge base was developed by:– Here the name of the knowledge base developer(s) is(are) recorded.
  • Acknowledgements: here all acknowledgements are listed – to those, other than the developers, who assisted in the design and creation of the knowledge base and to any funding bodies that supported the project.
  • Purpose - If the knowledge base was built as part of a particular project, or designed to supplement work in a certain field, this can be recorded here.
  • Methods – Here information about the methods used for selecting the interviewees and collecting the knowledge may be described.
  • Location – Here information about the exact location of the communities visited can be recorded.
  • Timing of Knowledge Collection – The time of year, and the season that the knowledge collection was undertaken, may influence the knowledge elicitation process and should therefore be noted.
  • General Comments – Here any other pertinent pieces of information may be recorded.

Object hierarchy

Panel

The main part of the webAKT window is occupied by the workspace. This is initially a large blank area, underneath the menus and toolbar at the top of the window. Just about everything you do in webAKT involves displaying panels in the workspace. Think of each panel as a small window (in fact, they were called "windows" or "dialogue windows" in AKT5).

Each panel can be used to:

  • display information about the knowledge base that the user is interested in, and/or
  • allow the user to enter information or make some choice - a dialogue window.
The same panel can server both purposes. For example, the statement_collection panel can display a set of statements, optionally filtered by the user; or can allow users to select an individual statement that they would like more detailed information on. Similarly, the statement_details panel can be used to display everything that is known about a particular statement; to enable the user to add to or modify this information; or to select one of the sources for this statement to get more information on that source.

There are 3 main types of panel:

  • The collection type of panel presents all the items in a particular AKT collection. For example, the statement_collection panel shows all the statement items in the knowledge base - the statement collection. This always has the name xxx_collection, e.g. statement_collection.
  • The details type of panel gives more details about a particular item in a collection. For example, the statement_details panel shows all the properties of a particular statement. This always has the name xxx_details, e.g. statement_details.
  • Separately, there can be panels associated with various tools available in your version of webAKT. These tools can be built in, provided by other people, or written by yourself. Each too undertaks some form of analysis of the knowledge base which is not catered for by the standard xxx_collection and xxx_details panels. For example, the species_report tool (a built-in tool) provides comprehensive information on a chosen species, including list of actual and derived statements that the species is associated with. There is no standard way of naming these panels.

It is quite common for users to create multiple panels of a particular type during a single webAKT session. For example, you could open up several statement_details panels, one for each of several statements selected from the statement_collection panel. In order for you to track these individually, the name of each panel is actually made up of its type (e.g. statement_collection) followed by a number,which indicaes the order in which it was created. For example, statement_collection_3 panel is the 3rd instance of the statement_collection panel created durng this particular webAKT session.

Please note an important distinction between AKT5 and webAKT. In AKT5, there could only be one instance of a panel type open at any one time. If you performed somee operation that displayed the same type of information, it would re-use the existing panel of thattype. In webAKT, the default is that a new panel instance will be created each time you want to display a ceertain type of panel (e.g. statement_details). This is useful for showing e.g. the detains of several statements on the screen at the same time, for side-by-side comparison etc. But it does mean the the screen can rapdily become cluttered with panels, so you will probably want to close panels (using the X) from time to time.

All panels are created by the same code, and have a number of features in common. These include:

  • A title bar, giving its name and the knowledge base (in brackets) that the information in the panel came from.
  • A Delete button (X). This permanently deletes the panel, and removes its name from the list of panels at the top of the workspace. Note that its sequence number (e.g. statement_collection_3) will not be re-used during this session.
  • A Minimise button (_). This hides the panel so it is not visible in the wokspace, butthe panel still exists: you can open it up again by clicking on its name in the list of panels at the top of the workspace.
  • A Settings button (cogwheel). This will allow you to set various setting that are specific to this panel, though currently this is not operational for any of the panel types.
  • Resizability. All panels can be resized by dragging the bottom right corner, th eright-hand side, or the bottom. Currently, this is not particularly useful, but eventually it will allow the contents to re-size as well.
  • Draggability. Panels can be dragged around the workspace by dragging the title bar.
  • A panel which is partly hidden behind other panels can be brought to the front (or raised to the top depending on how you want to think about it), by clicking anywhere in the visible part of the panel.

Source

Statements need to be attributed to one or more sources. This is essential for strengthening the credibility of individual statements. It also allows the statement to be checked with the original source later on if, for example, it turns out to be inconsistent with a statement made by another source. Further, it enables the investigator to exolore the relationship between individual sources and the statements they make.

webAKT, like AKT5, allows for two types of source:

  • An individual(typically, a local farmer) who provides the statement in the course of an interview; or
  • A published source, such as a journal paper.
The properties of a source are different for these two cases:
For a person, it includes: the source's name, sex, age and ethnic origin; the interviewer's name; and the date and location of the interview.
For a published source, it includes the publication's author(s), title, journal, volume etc

AKT5 obliged the knowledge acquisition person to associate a statement with a source, and would not allow the statement to be added to the current knowledge base until this was done. webAKT removes this obligation (though this decision could be reversed if required), but does allow the user to identify all statements that have not as yet had a source attached to them.

Synonym

In a knowledge base, there may be several words used for the same underlying concept, such as the local, English and scientific names for a plant, and even different local names. In order to cater for these, AKT5 (and hence webAKT) supports the notion of synonyms.

From the AKT Manual:

The ‘synonyms’ option allows you to specify a synonym for a formal term. An example of this might be the local name for a plant beside the common English name and the scientific Latin name. For example, the tree with the common English name ‘shea butter tree’ has the scientific name Vitellaria paradoxa, and is also commonly known in English by its French name ‘karité’ whilst in the Hausa language in Nigeria it is known as ‘ka’danya’, in the Bambouk language in Senegal it is known as ‘toulou’. Thus a knowledge base developed on the shea butter tree in West Africa would need to record all the common synonyms for the tree.

The synonym could also be for a value such as ‘decrease’ for ‘decline’ or a process such as ‘furrowing’ for ‘ploughing’.

The reason it is important to have the synonyms option is that it permits you to draw on all available statements using the formal term or any of its synonyms and prevents accidental exclusion in a topic knowledge base because different terms have been used for the same object/process/value etc.

However, there are two major issues with synonyms in AKT5. The first is that synonyms are “second-class citizens”: they are not listed in the glossary of formal terms, and they are not apparent when you look at a statement. The second is that synonyms do not have a type (e.g. “english_name”, “scientific_name”). (They can be given a numeric order in the list of synonyms for a formal term, and that can be used as a sort of typing mechanism, but that is an obscure way of achieving this goal.)

Both of these limitations are being addressed in webAKT. This is still work in progress, but it will allow you to (for example) ask for a list of statements using the scientific name where possible; or mouse-over the terms in a statement to see the synonyms for each term; or see all synonyms listed when you ask for a list of formal terms.

Deleting a formal term

These added terms may also be deleted by highlighting them in the ‘Formal Terms’ list and selecting Delete. However, it is not possible to delete a formal term that still appears in a formal statement. In order to remove a formal term which appears in a formal statement, the term must either be replaced, using the edit function in the Statements menu (see above, 7.4.4) or by deleting the statement altogether, using the Delete function in the Statements menu (see above 7.4.5).

The first 3 buttons call up the same panel, the formal_term_details panel, the only difference being that it’s appearance and the operations you can perform are slightly different for each of the 3 modes.

formal_term_details

The formal_term_details panel can open in one of 3 modes (New, View or Edit), depending on the action the user took which led to it being opened. Typically, the user would have been visiting the formal_term_collection panel, and clicked one of the 3 buttons New, View or Edit. There are small but significant differences in appearancedepending on the mode, as explained below.

Mode: New. Creating a new formal term

In New mode, the user can edit all the fields, and there is an Update button, but all the fields are initially empty.

You will probably not find yourself using this mode for formal terms. The reason is that, normally, formal terms are created automatically, when you enter a statement, as described in the documentation for the statement_details panel. webAKT does this by analysing the grammatical structure of the statement to determine the grammatical type of each term (e.g. object, attribute, process…), and adds it to the formal terms glossary. You can subsequently edit this (see the text on the Edit mode).

However, the New mode for this panel allows you to add new formal terms, unattached to statements, to the formal term glossary. This is particularly useful for objects and object hierarchies, since you might want to build up a glossary of common terms before, or in parallel with, entering statements.

First, check the list of formal terms to see whether this term, or a suitable equivalent, already exists in the formal term glossary. You do this by openig u the formal_term_collection panel (KB > Formak terms). If it doesn’t, click on the New button. This opens up a formal_term_details panel, and you can then provide a name, description, etc, and specify its type.

Once you are satisfied with the entry, do not forget to click the Update button to add the new formal term to the knowledge base.

Mode: View. Viewing a formal term

The View mode allows you to see details of a formal term without being able to change them - there is no Update button, and you cannot edit any of the fields. This could be useful if you want to make sure that you do not inadvertingly make some change. This mode could alos be useful if you want to allow the user to explore a knowledge base without being able to make any changes - e.g. in a teaching context - in whoch case a global setting forView mode could be set.

The panel displays the formal term type, its definition and any synonyms there may be.

If you click on the Statements button, the statement_collection panel appears, showing all the statements in which the formal term appears.

If you click on the Hierarchies button, and the formal term refers to an object and it's in a hierarchy, then a hierarchy_details panel appears for the relevant hierarchy, showing the formal term's position in the hierarchy.

Viewing, creating and editing statements

The core of an AKT knowledge base is its collection of statements. This section describes the user interface in webAKT for entering new statements, and for viewing and editing statements. It does not go into the rules for formulating statements according to the AKT grammar: that is quite involved, and is covered in a separate document.

Having said that, webAKT does provide a “wizard” to make it easier to formulate statements, in the form of a set of templates matching AKT’s statement grammar, with slots for particular nouns in the statement. This accessed via the Template button in the panel used for creating a new statement. So even if you have not acquainted yourself with AKT’s grammar, you can still get started by entering made-up statements which are syntactically correct (i.e. which conform with AKT’s grammar), to familiarise yourself with the procedure for entering statements.

Displaying all statements

The statements in the knowledge base are displayed in the Statements panel, which is invoked using the KB > Statements command. If you are starting a new knowledge base, this panel will be empty as in Figure XXX. If you are adding statements to an existing knowledge base, the statements previously entered will be listed in the ‘All Statements’ listbox as in Figure XXX.

The information shown in this listbox is customizable. Currently, it shows the statement ID, and the formal and quasi-English versions of each statement. This will be extended with options to allow the user to choose which properties of each statement should be displayed.

Filtering the statements

webAKT provides a simple yet powerful method for filtering the statements (i.e. selecting a subset of them according to user-defined criteria). This makes use of a mixture of checkboxes and listboxes.

The five checkboxes relate to the type of statement, and statements are included if the corresponding checkbox is ticked. The types are:

  • att_value, causal or comparison statements; and
  • non-conditional or conditional statements.

The three listboxes relate to particular properties of the statement, and statements are restricted to those that satisfy the specified criteria, i.e. that you selec from the listbox menu. The three criteria are:

  • statements that contain a specified formal term;
  • statements that were provded by a specified source; and/or
  • statements that are relevant to a specified topic.
If one or more of these filters is left unchecked, then that criterion does not apply.

Note that the filtering is applied immediately, as soon as you check or uncheck a checkbox, or change the specified restriction value. Working with individual statements

The Statements panel provides 4 buttons for working with individual statements:

  • New: Opens up a new panel for entering a new statement.
  • View: Opens up a new panel for viewing (read-only) the details of a selected statement. The statement must first be selected from the listbox.
  • Edit: Opens up a new panel for editing the details of a selected statement. The statement must first be selected from the listbox.
  • Delete: Deletes a selected statement. The statement must first be selected from the listbox.
  • Select all: Selects all statements in the statement listbox.
  • Deselect all: Deselects all statements in the statement listbox.
  • Invert: Inverts the selection status of each statement in the statement listbox.

The first 3 open up the same panel type – a ”statement_details” panel. The exact contents of this panel depends on which mode (new, view or edit) you choose. See the Reference Manual page [[provide link]] for the statement_details panel for more information on these three modes.

Note that the selection buttons only apply to the statements in the listbox, i.e. after filters have been applied - not to all the statements in the knowledge base. So doing "Select all" after applying one or more filters is a powerful way of selecting all statements that satisfy some set of filters. If you then clear the filters, then those statements will remain selected in the list of all statements.

Creating a new statement

In order to enter a new statement select New in the Statements panel

Figure 7.11 is an example of a newly entered statement.

Figure 7.11 The ‘New Statement’ dialog box

All unitary statements have to be entered using the formal language (AKT Manual, Chapter 4.2), in the ‘Formal Language Statement’ box. As you type the statement, the syntax of the statement is checked character-by-character. If it is incomplete or incorrect, then a red bar appears at the right of the edit box. If the statement entered so far is syntactically correct (according to the AT grammar), then the bar changes to green.

Note the statement may be syntactically correct even if you have not finished it, e.g. if you are going to add a conditional clause (starting with “if”) after the main part of the statement.

The formal statement must conform to the grammar described in the AKT Manual, chapter 4.2. If you receive any error messages then either the structure of the statement does not conform to the formal grammar, or one of the reserved terms (e.g. att_value) is misspelt.

Translating the statement into English

Once the formal language statement has been correctly entered, select Translate from the ‘New Statement’ dialog box for a natural language translation of the formalized statement. The natural language statement generated (Figure 7.14) will be stylised and will not always read smoothly or be grammatically correct. It should, however, make sense and express the meaning of the unitary statement unambiguously.

When you are satisfied with the statement, press Update. Please note that this only updates the current knowledge base in the webAKT application. The programme automatically adds it to the updated ‘Statements’ panel (Figure X) from which all existing statements can be accessed. To save it in the knowledge base permanently you must select Save as under the File menu.

Linking words in the statement to the glossary of formal terms

If your statement includes any formal terms new to the knowledge base, webAKT will automatically determine the type (grammatical role) of the formal term from its position in the statement, and will automatically create a new formal term with this name and type. Associating a statement with one or more sources

In webAKT, you may, but do not have to, associate a statement with a source when you create the statement. This is a marked departure from AKT5, where you had to do this with one source before you could add the statement to the knowledge base. (If you wanted to include more than one source, you had to do that as a separate operation.)

The reason for this decision is to avoid interrupting the flow of adding statements by forcing the knowledge base creator to divert and create a new source. Sources can always be added afterwards, and indeed webAT easily allows you to find all statements that do not have a source specified.

If you do not like this approach, then you can always add information about your sources first: then associating them with a statement is easy. (Alternatively, you can complain, and webAKT can be changed to emulate the AKT5 behaviour.)

To add associate a statement with a source that already exists, click on the Add source button. If the Sources panel is not already open, then it will appear. Then click on the relevant entry in the Sources panel listbox, and it will be added in the Source listbox for this particular statement. You can then click on additional sources if they have also expressed this statement.

To detach a source from a statement, click on the Detach source button, then click on the relevant source in the Sources listbox for this particular statement (not in the listbox in the main Sources panel).

Summarising the contents of the knowledge base

This command, new to webAKT, in the KB menu provides a quantitative summary of the various components of the knowledge base – for example, the number of statements and sources. The headings are the same as in the KB menu itself, and each heading provides a link to the same Panel accessed through the KB menu.

Like many other panels, this panel is “live”, meaning that it updates automatically whenever the knowledge base is changed, for example by adding a statement.
